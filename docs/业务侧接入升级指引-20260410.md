# 资产浏览器 SDK 升级指引 — 2026-04-10

## 本次变更：支持宿主接管草稿发布流程

### 背景

此前 `AssetBrowserWorkspace` 已提供 `onBeforePublishDraft` 和 `onAfterPublishDraft` 两个发布相关回调，宿主应用可以在默认发布前后插入业务逻辑。

但在真实业务接入里，发布动作往往不是单纯地“把当前 draft 发布掉”。常见场景是宿主系统需要先完成一段业务编排，再把最终生成的业务版本号作为 `versionId` 传给资产浏览器的发布接口。

典型链路如下：

1. 业务系统根据当前状态生成新的业务版本号，例如 `vYYYYMMDDHHMMSS.hash`
2. 把这个版本号回写到业务数据库或审批系统
3. 再调用资产发布接口，并把该业务版本号作为 `versionId` 传入
4. 发布成功后，把工作台切换到新版本，并清理草稿编辑态

旧版本的问题不在于“不能拦截默认发布”，而在于“宿主自己发布完成后，缺少公开方式优雅地刷新工作台状态”。很多接入方只能通过重挂载组件来强制刷新 UI。

### 本次新增能力

`onBefore...` 回调上下文里的 `workspaceActions` 现在正式作为公开能力提供，宿主在拦截默认发布后，可以调用这些动作驱动工作台状态同步。

当前提供的动作：

| 动作 | 类型 | 说明 |
|------|------|------|
| `refreshWorkspace` | `() => Promise<void>` | 重新拉取 collection、versions 和当前版本树；若当前选版仍有效，会尽量保留 |
| `selectVersion` | `(versionId: string) => void` | 切换到指定业务版本 |
| `clearDraftState` | `() => void` | 关闭 Diff、清理脏状态、清理编辑器缓存和标签页 |

这意味着宿主不再需要通过 remount 整个工作台来刷新发布后的界面状态。

### 推荐接入方式

推荐直接在 `onBeforePublishDraft` 中接管发布：

1. 宿主完成业务编排
2. 宿主调用自己的后端发布 draft，并传入业务版本号
3. 调用 `refreshWorkspace()` 获取最新 collection 和版本列表
4. 调用 `selectVersion(newVersionId)` 切换到新发布版本
5. 调用 `clearDraftState()` 清理草稿态
6. 返回 `false`，阻止 SDK 默认发布流程继续执行

示例：

```tsx
<AssetBrowserWorkspace
  client={client}
  assetSpace="skills"
  assetId={skillId}
  callbacks={{
    onBeforePublishDraft: async (context) => {
      const actions = context.workspaceActions;
      if (!actions || !context.draftVersionId) {
        return false;
      }

      const businessVersionId = await createBusinessVersionId();

      await fetch(`/api/skills/${context.assetId}/publish`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          draftVersionId: context.draftVersionId,
          versionId: businessVersionId,
        }),
      });

      await actions.refreshWorkspace();
      actions.selectVersion(businessVersionId);
      actions.clearDraftState();
      return false;
    },
  }}
/>
```

### 为什么现有回调现在够用了

核心变化不是新增了另一套发布回调，而是让现有 `onBeforePublishDraft` 拿到了正式的工作台控制动作。

这样做有几个好处：

1. 与现有回调语义兼容，不破坏旧接入
2. 业务发布返回体仍由宿主自己定义，SDK 不需要理解业务编排结果
3. 宿主可以在同一个回调里完成“拦截默认发布 + 自定义发布 + 状态回填”整条链路
4. 发布成功后的 UI 刷新从“旁路 remount”变成“受支持的状态同步动作”

### 行为边界

- `versionId` 仍然由宿主系统决定，SDK 不负责生成业务版本号
- 如果 `onBeforePublishDraft` 没有返回 `false`，SDK 会继续执行默认发布
- `onAfterPublishDraft` 仍只在 SDK 默认发布成功后触发
- 未使用 `workspaceActions` 的宿主，现有行为保持不变

### 升级建议

如果你当前已经通过“拦截发布 + remount 工作台”实现宿主接管，可按下面方式收敛：

1. 保留原有业务版本号生成和发布 API
2. 删除发布成功后的组件 remount 逻辑
3. 改为在 `onBeforePublishDraft` 内调用 `refreshWorkspace()`、`selectVersion()`、`clearDraftState()`

升级后，工作台可以直接切换到宿主刚发布出的版本，且不会残留旧 draft 的 Diff 和编辑状态。
