# 业务侧接入升级指引（2026-04-07）

面向对象：准备接入或正在升级业务资产浏览前端的业务团队。

这次更新的重点，不是新增一组零散组件，而是把 preview 里那套已经验证过的业务控制台页面效果，正式沉淀成可发布、可复用的 SDK 能力。

本次推荐的接入顺序很明确：

1. 默认优先使用 `AssetBrowserConsoleWorkspace`
2. 如果你已经在用 `AssetBrowserWorkspace`，优先改成 `appearance="console"`
3. 只有在你要完全接管页面内部编排时，才使用 `AssetBrowserConsoleShell`
4. 不再建议业务项目直接复制 `preview/main.tsx`

## 本次更新带来了什么

业务前端现在可以直接复用以下能力，而不需要自己再拼一层页面壳：

- 中文控制台式顶部栏、状态条和左右布局
- 左侧资源目录区内置搜索、版本说明和导出入口
- 右侧统一的“编辑 / 预览 / 差异”切换
- 移动端自动收敛为上下布局
- Preview 已经和运行时工作台复用同一套 console shell，后续样式和版式不会再分叉维护

当前可直接使用的公开入口有：

- `AssetBrowserWorkspace`
- `AssetBrowserConsoleWorkspace`
- `AssetBrowserConsoleShell`
- `AssetTree`
- `AssetEditor`
- `AssetDiffViewer`

## 依赖安装方式

这两个包现在按独立仓库使用，业务项目应当分别安装：

```json
{
  "dependencies": {
    "protobuf-typescript-client-gen": "github:ahaeureka/protobuf-typescript-client-gen",
    "stew-asset-browser-react": "github:ahaeureka/stew-asset-browser-react"
  }
}
```

不再推荐以下方式：

- 在业务项目里用 `workspace:*` 指向这两个包
- 用 `link:` 假设业务项目和 SDK 项目永远在同一个本地工作区
- 只安装 `stew-asset-browser-react`，却不显式安装 `protobuf-typescript-client-gen`

原因是这两个包是独立交付单元：

- `protobuf-typescript-client-gen` 负责提供浏览器端 client 和协议层能力
- `stew-asset-browser-react` 负责提供资产浏览 UI SDK

业务项目通常会同时直接依赖它们，而不是通过 workspace 绑定在一起。

## 业务侧如何选

### 方案 A：新页面直接接入

适用场景：

- 你还没有落地资产浏览页
- 你想直接拿到一套接近 preview 的可用页面
- 你不想自己再拼目录树、编辑器、Diff、版本和导出动作

推荐方案：使用 `AssetBrowserConsoleWorkspace`

```tsx
"use client";

import { useMemo } from "react";
import { AssetBrowserClient } from "protobuf-typescript-client-gen/dist/asset_browser_client";
import { AssetBrowserConsoleWorkspace } from "stew-asset-browser-react";

export default function AssetConsolePage() {
  const client = useMemo(
    () => new AssetBrowserClient({
      baseUrl: "http://localhost:3012",
      timeout: 15000,
    }),
    [],
  );

  return (
    <AssetBrowserConsoleWorkspace
      client={client}
      assetSpace="configs"
      assetId="gateway-routing"
      title="网关路由资产中心"
      height="calc(100vh - 96px)"
    />
  );
}
```

### 方案 B：已有 AssetBrowserWorkspace 页面升级外观

适用场景：

- 你已经接了 `AssetBrowserWorkspace`
- 你当前主要诉求是切到 preview 的产品化外观
- 你不想改动现有数据流和事件回调

推荐方案：在现有接法上增加 `appearance="console"`

```tsx
<AssetBrowserWorkspace
  client={client}
  assetSpace="configs"
  assetId="gateway-routing"
  appearance="console"
/>
```

这是本次升级成本最低的路径。

### 方案 C：你要保留自己的数据流，但想复用页面壳

适用场景：

- 你的目录树、编辑区、Diff 来自自研状态管理或 BFF
- 你只想复用 console 布局和视觉风格
- 你不接受 `AssetBrowserWorkspace` 的默认行为边界

推荐方案：使用 `AssetBrowserConsoleShell`

这时你需要自己提供：

- `controls`
- `actions`
- `sidebarContent`
- `mainContent`
- 可选的 `viewSwitcher`、`compareNote`、`footer`

简单理解：

- `AssetBrowserConsoleWorkspace` 是成品工作台
- `AssetBrowserConsoleShell` 是页面壳

## 不建议的做法

以下做法本次升级后都不再推荐：

- 直接复制 `preview/main.tsx` 到业务项目中继续改
- 手工拷贝 preview 的 CSS，再自己维护一套同名布局
- 为了接 preview 风格，重新拼一套目录树 + 编辑器 + Diff 组合
- 在业务代码里深引 preview 目录下的文件

原因很简单：preview 现在已经降级为演示消费者，而不是业务项目应该依赖的公开 API。

## 业务侧最小升级步骤

### 从 0 到 1 接入

1. 在业务项目里分别安装 `protobuf-typescript-client-gen` 和 `stew-asset-browser-react`
2. 在页面里创建 `AssetBrowserClient`
3. 优先挂 `AssetBrowserConsoleWorkspace`
4. 传入 `assetSpace`、`assetId`、`height`
5. 根据页面需要再补 `title`、`initialVersionId`、`initialFolder`

### 从旧页面切到 console 外观

1. 保留原有 `AssetBrowserWorkspace` 调用
2. 增加 `appearance="console"`
3. 检查原有 `renderToolbarStart`、`renderToolbarEnd`、`renderFooter` 是否仍符合新版页面布局
4. 校验页面在桌面和移动端下的高度设置

### 从“复制 preview”迁回 SDK 公共入口

1. 删除业务项目里复制的 preview 壳子 JSX
2. 如果你的目标是成品页，直接替换成 `AssetBrowserConsoleWorkspace`
3. 如果你只想留自己的内部状态管理，替换成 `AssetBrowserConsoleShell`
4. 把原来散落在业务侧的目录树 / 编辑器 / Diff 逻辑，收回到 SDK 公开组件组合上
5. 删除业务项目里冗余的 preview 样式副本

## 这次升级对后端能力的依赖

业务侧如果要完整用好当前工作台，默认应假设后端已经支持以下行为：

- 根目录 `initialFolder="/"` 返回整版资产的递归平铺条目
- `getVersion()` 可返回 `baseVersion`
- draft 版本可返回 `draftDiffSummary`
- `diffDraft()` / `diffVersions()` 已实际支持 `diffMode`、`pathPrefix`
- `diffMode="with_text"` 时可返回 `oldPreview`、`newPreview`、`unifiedDiff`、`textDiffStatus`
- 目录导出支持打包 zip

如果你接入后发现差异区仍在大量前端拼装数据，通常不是 UI SDK 入口选错，而是后端接口能力还没有对齐到当前版本。

## 业务侧上线前检查

建议至少检查以下 8 项：

1. 页面是否已经从公开入口导入，而不是深引 preview 文件
2. `AssetBrowserClient` 是否从 `protobuf-typescript-client-gen/dist/asset_browser_client` 导入
3. 页面是否优先使用 `AssetBrowserConsoleWorkspace`，而不是重复拼装壳子
4. 根目录树是否能正确展示目录节点，而不是只显示零散文件
5. Markdown 文件是否能在“编辑 / 预览”间切换
6. 差异区是否能直接看到后端返回的文本 diff 结果
7. 导出文件和导出目录是否都能工作
8. 页面在窄屏下是否仍可用

## 常见决策建议

如果你在这三个方案之间摇摆，可以按下面判断：

- 只想尽快上线：选 `AssetBrowserConsoleWorkspace`
- 已经接了默认工作台，只想换风格：选 `AssetBrowserWorkspace + appearance="console"`
- 已有复杂页面框架，只想借用控制台外观：选 `AssetBrowserConsoleShell`

默认情况下，不建议业务团队从 `AssetBrowserConsoleShell` 开始接。那会把你重新带回“自己维护页面编排”的复杂度里。
